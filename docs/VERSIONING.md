# Версионирование Scenaria

Проект следует [Semantic Versioning 2.0.0](https://semver.org/lang/ru/) в формате **MAJOR.MINOR.PATCH** (например `0.3.2`).

## Источник правды

| Место | Роль |
|-------|------|
| `pyproject.toml` → `version` | Единственная версия, которую меняем вручную перед релизом |
| Git-тег `vX.Y.Z` | Должен совпадать с `pyproject.toml` (префикс `v` только в теге) |
| `dist/Scenaria/version.txt` | Записывается скриптом сборки из `app_version()` |
| GitHub Release | Артефакты и `latest.json` для автообновления |

Проверка в коде: `app.version.app_version()` читает bundled `version.txt` или `pyproject.toml`.

## Пока версия `0.x` (до 1.0.0)

Scenaria ещё в стадии **0.x** — API и UX могут меняться. Договорённость для этой линии:

| Часть | Когда увеличивать | Примеры |
|-------|-------------------|---------|
| **PATCH** (`0.2.11` → `0.2.12`) | Исправления багов, мелкий UX, тесты, CI, документация. Без новых пользовательских возможностей. | фикс hover при воспроизведении, стабильность панели, падение рекордера |
| **MINOR** (`0.2.x` → `0.3.0`) | Новая **заметная** возможность, обратно совместимая с `.feature` и сценариями на диске | HTTP Basic Auth для сайтов, вкладки редактора, пакетный запуск |
| **MAJOR** (`0.x` → `1.0.0`) | Стабильный релиз «1.0»: готовность поддерживать формат сценариев и публичное поведение; или **намеренно ломающие** изменения | смена синтаксиса Gherkin, удаление меню, другой формат настроек |

**Правило большого пальца:** если в release notes есть блок «Новое» с пунктами для пользователя — чаще **MINOR**. Если только «Исправлено» — **PATCH**.

### Примеры из истории

| Релиз | Логичная ступень | Почему |
|-------|------------------|--------|
| 0.2.6 → contextual click, overlay | PATCH | доработка воспроизведения |
| 0.2.7–0.2.11 (auth, вкладки, welcome, hover) | лучше было **0.3.0** | несколько новых фич в одном выпуске |
| 0.2.12 (гипотетически: только фикс CI) | PATCH | инфраструктура |
| 0.8.0 (TestClient, единый GUI) | MINOR | новая возможность для пользователя |

Опубликованные номера не переименовываем.

## После 1.0.0

- **PATCH** — багфиксы, без изменения задокументированного поведения.
- **MINOR** — новые функции, старые сценарии и файлы работают.
- **MAJOR** — ломающие изменения; миграция или явное предупреждение в release notes.

## Процесс релиза (кратко)

1. Выбрать следующую версию по таблице выше.
2. Добавить секцию в [CHANGELOG.md](CHANGELOG.md) (перенести пункты из `[Unreleased]`).
3. Обновить `version` в `pyproject.toml`.
4. Закоммитить: `Release vX.Y.Z: краткое описание.`
5. Тег и push:

```powershell
git tag vX.Y.Z
git push origin master
git push origin vX.Y.Z
```

5. Дождаться успешного workflow [Release](https://github.com/bafgion/scenaria/actions) — он соберёт ZIP и опубликует GitHub Release.

## Чеклист перед тегом

- [ ] Версия в `pyproject.toml` совпадает с планируемым тегом
- [ ] `python -m pytest tests/ -q` проходит локально
- [ ] Release notes понятны пользователю (новое / исправлено)
- [ ] Не плодить несколько тегов из‑за CI: сначала зелёные тесты, потом один тег

## Автообновление

Приложение сравнивает `app_version()` с `tag_name` на GitHub (`v0.3.0` → `0.3.0`). Пользователю показывается более новая версия по semver; downgrades не предлагаются.
